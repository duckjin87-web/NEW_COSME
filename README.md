# NEW_COSME

신규 화장품 제조업체 방문 전, 여러 카테고리(CAPA·설비 / 재무·신용리스크 / 행정처분·리콜 / 거래처·브랜드 / 특허 / 인증·기술력 / 근황·기사)를 통해 신뢰도 있는 사전정보를 모으는 대시보드입니다.

## 사용법

`index.html`을 브라우저로 열면 등록된 업체 목록이 검색창과 함께 표시됩니다. 업체를 선택하면 카테고리별 데이터, 출처/신뢰도, 종합 신호등, 방문 시 확인 체크리스트가 표시됩니다.

```
python3 -m http.server 8000   # 저장소 루트에서 실행
# http://localhost:8000/
```

GitHub Pages로도 그대로 서빙됩니다 (`https://<user>.github.io/<repo>/`).

## 구조 (데이터 / 템플릿 분리)

```
data/
  categories.js        # 모든 업체 공통 카테고리 스키마 (1회만 정의, 수정 거의 불필요)
  registry.js           # registerCompany() — 회사 데이터 등록 함수
  companies/
    manifest.js          # 등록된 업체 id 목록
    eson.js               # 업체별 데이터 파일 (예시: (주)이손)
assets/
  dashboard.js          # 카테고리 스키마 + 회사 데이터만으로 화면을 그리는 공용 렌더링 엔진
  dashboard.css
index.html              # 단일 진입점 (목록 ↔ 상세, ?biz=<id> 로 라우팅)
```

## 신규 업체 추가 방법

1. `data/companies/<id>.js` 생성, `registerCompany({...})` 호출 (`data/companies/eson.js`를 템플릿으로 복사)
2. `data/companies/manifest.js`의 `COMPANY_FILES` 배열에 `<id>` 추가
3. 카테고리별로 공개 자료에서 확인한 내용만 `rows`/`badgeGroups`/`items`에 채우고, 모든 값은 `evidence` 객체에 출처·원문·신뢰도(`high`/`mid`/`low`)와 함께 기록
4. 공개 자료로 확인 불가능한 값(예: 월 CAPA, MOQ, 단가)은 비워두면 자동으로 "방문 시 확인 체크리스트"에 포함됨
5. 동명/관계사 혼동 리스크가 있으면 `aliasWarning`에 기록 — 상세 화면 상단에 경고 배너로 표시됨

### 근거자료 수집 방법 (카테고리별)

| 카테고리 | 주요 출처 |
|---|---|
| CAPA·설비 | 채용공고(잡코리아/사람인), 회사 홈페이지, 방문 실측 |
| 재무·신용리스크 | DART(dart.fss.or.kr) 공시, NICE평가정보(로그인 필요 — 반자동) |
| 행정처분·리콜 이력 | 식약처 행정처분 통합공고(mfds.go.kr) — 회사명/소재지로 직접 검색 |
| 거래처·브랜드 포트폴리오 | 업계 매체 인터뷰, 채용공고 내 거래처 언급, 전시회 참가이력 |
| 특허·지식재산권 | KIPRIS(kipris.or.kr) 출원인 검색 |
| 인증·기술력 | 회사 홈페이지, 채용공고 연혁, 보도자료 |
| 근황·기사 | 코스모닝, 인코스메틱스 코리아 등 업계 매체 |

DART처럼 공개 API가 있는 출처는 추후 자동 연동을 검토하고, NICE·KIPRIS·식약처처럼 로그인/직접 검색이 필요한 출처는 위 절차대로 수동 조사 후 입력합니다.

### 공공데이터포털(식약처) API 연동 검토 결과

"업체명으로 행정처분/업체정보를 자동 연결할 수 있는가"를 검토한 결과:

- **가능**: `식품의약품안전처_화장품 제조업체 정보` (data.go.kr, 서비스 `CsmtcsMfcrtrInfoService01`)로 업체명 조회 시 **사업자등록번호·대표자명·주소·등록일**을 받을 수 있습니다. 이는 위 "거래처·브랜드" 카테고리가 아니라 `bizNo`(사업자등록번호) 확정 — 즉 **동명/관계사 오인 리스크 해소**에 직접 쓸 수 있는 근거입니다.
- **불가능**: 화장품 **행정처분·리콜 이력** 전용 공개 API는 없습니다 (식약처는 의약품 행정처분 API만 제공). 따라서 `sanction` 카테고리는 계속 mfds.go.kr 행정처분 통합공고 게시판을 수동 검색해야 합니다.
- **연동 제약**: (1) data.go.kr에서 활용신청 후 발급받는 `serviceKey`가 필요 — 본 저장소 작업자가 임의로 발급할 수 없으므로 사용자가 직접 발급해야 합니다. (2) 이 사이트는 백엔드 없는 정적 GitHub Pages이므로, 브라우저에서 곧바로 API를 호출하면 CORS로 차단될 가능성이 높습니다.
- **현재 상태**: 미연동(설계만 문서화). `serviceKey` 확보 시 다음 순서로 적용 가능합니다.
  1. (우선) 데이터 등록 시점에 조사자가 API를 직접 호출해 `bizNo`/주소/대표자를 확인하고, 그 결과를 `data/companies/<id>.js`에 검증값 + `evidence`로 반영 (자동 호출·수동 반영).
  2. (선택) 사이트 내 실시간 업체명 검색까지 필요하면 별도 서버리스 프록시(Cloudflare Worker 등)를 추가해 CORS를 우회 — 현재 범위 밖, 별도 설계 필요.

## 종합 신호등 로직

`assets/dashboard.js`의 `computeSignal()`이 카테고리별 근거 확보 여부를 종합해 판정합니다.

- 🔴 리스크 확인됨 — 행정처분/리콜 이력에 위반 사실 확인 시
- 🟢 정보 충분 — 카테고리 70% 이상 근거 확보 + 재무 확인됨
- 🟡 부분 확인 — 카테고리 40% 이상 근거 확보
- ⚪ 정보 부족 — 그 미만
